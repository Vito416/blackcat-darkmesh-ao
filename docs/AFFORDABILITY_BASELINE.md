# Affordability Baseline Method (Draft)

Status: draft for legal, product, and policy review

This document explains the intended method for estimating the affordability baseline referenced by the draft Founder Fee Policy.

## 1. Purpose

The purpose of this document is to make the Accessibility Principle more concrete by defining a repeatable way to estimate:

- the baseline first-year direct operating cost of a comparable small operator,
- the affordability threshold used for ordinary fee schedules,
- and the public evidence used to support schedule revisions.

## 2. Baseline operator profiles

Unless a more specific profile is published for a coverage class, the default baseline should be estimated using one of these profiles:

- `single-site operator`
- `single-shop operator`
- `single-gateway operator`

Each profile should reflect a minimal but realistic production deployment rather than an unrealistically stripped-down or enterprise-grade setup.

## 3. Included cost categories

The first-year direct operating cost estimate may include publicly observable market costs for:

- entry-level hosting or compute suitable for the profile,
- domain registration,
- TLS or certificate cost where not otherwise included,
- ordinary storage,
- ordinary bandwidth or traffic allowance,
- and comparable unavoidable baseline operational costs for a small production deployment.

It should not ordinarily include:

- custom development labor,
- premium enterprise support,
- bespoke integrations,
- optional analytics or premium add-ons,
- or extraordinary scaling costs not typical for the baseline profile.

## 4. Public pricing method

The intended default approach is to use public market pricing rather than private quotes.

Recommended method:

- gather at least three publicly observable price points from reasonably comparable providers for the relevant profile;
- normalize them to the same coverage period;
- exclude obvious outliers where justified;
- and use a median or similarly robust central measure.

## 5. Regional relevance

Where practical, affordability should be assessed with reasonable regard to the market most relevant to the target operator class.

Recommended good-faith approach:

- use a geographically relevant baseline where the deployment class is region-specific;
- use a more international baseline where the target class is globally distributed;
- and document the chosen market logic when publishing a schedule.

## 6. Default affordability threshold

Unless a more protective rule is published, the intended default affordability threshold remains:

- no more than 5% of the publicly observable baseline first-year direct operating cost of a comparable small operator for the relevant ordinary deployment class.

This is intended as a rebuttable ceiling for ordinary schedules, not a floor.

## 7. Evidence note for ordinary fee revisions

Where practical, an ordinary fee revision should be accompanied by a short public note identifying:

- the prior fee,
- the updated fee,
- the relevant deployment class,
- the inflation or value-preservation basis,
- the public affordability baseline used,
- and the conclusion that the new fee remains consistent with the Accessibility Principle.

## 8. Mandatory component treatment

In estimating affordability for an ordinary deployment class, mandatory official internal components should be treated as part of the same integrated system cost analysis.

The affordability baseline should not be manipulated by:

- excluding a materially necessary internal dependency,
- redefining a mandatory component as optional without genuine operational basis,
- or moving ordinary required functionality into separately priced packages in a way that hides the true cost burden.

## 9. Review cadence

The baseline method should be reviewed periodically, especially when:

- hosting markets materially shift,
- the intended operator base changes,
- or a successor value standard replaces the prior reference standard.
